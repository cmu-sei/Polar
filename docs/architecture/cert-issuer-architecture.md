# Cert Issuer: Architecture Specification

**Status:** Draft for review
**Implementation Language:** Rust
**Scope:** v1 (Kubernetes-only, OIDC-bootstrapped, no cosign verification)
**Successor to:** Attestation Service spec (deferred; this is a smaller, faster predecessor)

---

## 1. Purpose

The cert issuer is a service that mints short-lived X.509 client certificates for Polar agents at startup. Agents authenticate to the cert issuer using their projected Kubernetes service account tokens, and they receive a certificate signed by the Smallstep CA that other internal services (Cassini, the credential agent, and eventually Neo4j and other mTLS endpoints) accept.

This service exists to remove TLS material from agent deployment manifests. Today, every Polar agent has cert-manager-issued certs mounted into its pod via Kubernetes secrets, and every agent's manifest carries the configuration plumbing for those mounts. With sixteen agents across multiple environments and a Dhall-rendered manifest pipeline that has to keep all of this consistent, the per-agent TLS configuration is the largest single source of configuration drift in the system. The cert issuer collapses this into a single bootstrap call at agent startup: the agent presents its projected SA token, gets a certificate, and proceeds. Manifests stop carrying TLS material entirely.

The cert issuer is also the trust anchor for the credential agent that will be built after it. The credential agent's clients will authenticate via the certs this service issues; the credential agent itself will obtain its own cert from this service at startup. Building the cert issuer first means the credential agent has a clean authentication primitive to depend on rather than inventing one. This ordering is deliberate and not optional.

## 2. Scope and Non-Goals

This document specifies v1 of the cert issuer. v1 is intentionally minimal. It supports Kubernetes deployment only, validates a single OIDC issuer (the cluster's API server), performs no supply-chain verification on the requesting workload's image, and uses the `polar-nu-init` init container to perform the cert handshake on each agent's behalf. These limitations are deliberate simplifications, not oversights, and Section 12 documents the v2 directions that lift the security-relevant ones.

The init container is not a v2 item. It is the v1 client deployment model. The Polar agents do not all have HTTP clients in them, and adding one to each of sixteen agents purely to call the cert issuer is not a tradeoff worth making. The init container handles the handshake, writes the cert and key to a shared `emptyDir` volume, and exits; the agent reads the cert and key from the filesystem at startup, the same way it would read any other configuration file. This pattern also has a real security property — the SA token is only present during the brief init phase rather than for the agent's full lifetime — but the primary motivation for v1 is pragmatic: agents don't gain a new dependency they don't already have. The HTTP client lives in one place (the init container image), maintained once.

Out of scope for v1: cosign signature verification (a real security improvement, deferred until v2), bare-metal targets (deferred until there's a concrete bare-metal deployment to design against), multi-issuer OIDC abstraction (deferred until there's a second issuer to support), the challenge-response handshake from the original attestation protocol (a property worth having eventually but not necessary for v1's threat model), HMAC-bound ephemeral secrets (same), workload-image attestation via downward API (same), and credential distribution beyond TLS material (that's the credential agent's job and is a separate service).

What v1 *does* include: the `polar-nu-init`-based handshake, OIDC validation of projected service account tokens, CSR generation with the agent's identity in the SAN, certificate issuance via the Smallstep CA, structured event emission for every issuance attempt, Prometheus metrics, and Cassini topic publication of issuance events for the observability framework.

The v1 baseline is "strictly better than cert-manager mounts." That bar is meaningful: cert-manager mounts have no supply-chain verification either, no per-instance binding beyond the pod's identity, and no fresh issuance per agent restart. v1 of the cert issuer matches cert-manager on those properties (it's not worse) and adds short-lived certs requested at startup plus reduced bootstrap-credential exposure via the init container model (it's better on both axes). v2 raises the bar further with cosign and workload-image attestation, but v1's job is to ship and remove the configuration burden.

## 3. Architectural Constraints

The constraints from the original attestation service design carry forward, restated here for completeness because they shape the v1 design just as much as they shaped the larger one.

The cert issuer cannot depend on the Neo4j knowledge graph. The graph is populated by agents that depend on the cert issuer for their certificates; a graph dependency would create a bootstrap cycle. The cert issuer also cannot depend on the credential agent, for the same reason: the credential agent's clients will authenticate using certs from this service, so the credential agent depends on the cert issuer, not the other way around. Any secret material the cert issuer needs (the CA provisioner credential, eventually the cosign verification key in v2) must come from a source outside both of these — Kubernetes secrets, Vault, or KMS-backed envelope encryption.

The host is not a trust anchor. Pod identity comes from cryptographically signed assertions issued by the Kubernetes control plane (projected service account tokens), not from host introspection. This is the same property that ruled out SSH-based verification in the original spec and it carries forward unchanged.

Artifact stores are observed, not trusted. v1 does no cosign verification, so this constraint is not actively exercised in v1 — but it's worth restating because v2 will exercise it, and the v1 design should not preclude v2's adherence to it.

## 4. System Overview

The cert issuer is a single Rust binary deployed as a Kubernetes Deployment in a dedicated namespace with restrictive NetworkPolicy controls. It exposes an HTTPS API on a configured port for cert-issuance requests, and it depends at runtime on three external systems: the Kubernetes API server's OIDC issuer endpoint (for JWKS retrieval and JWT validation), the Smallstep CA (`step-ca`) for certificate issuance, and a secrets backend for the CA provisioner credential.

Clients of the cert issuer in v1 are `polar-nu-init` init containers running alongside each agent's workload container. The init container performs the cert handshake — reading the agent's projected service account token, generating a CSR, calling the cert issuer's `/issue` endpoint, and writing the resulting cert and key to a shared `emptyDir` volume — and exits cleanly. The agent's workload container starts after the init container exits, reads the cert and key from the shared volume, and uses them for mTLS to Cassini and any other internal service. The agent never makes an HTTP call to the cert issuer; the agent never holds the projected SA token in its own process. This is the deployment pattern for v1 and is described in detail in Section 7.

The cert issuer is integrated into the observability framework as a first-class telemetry source. Every issuance attempt produces a structured event published both to stdout (durable, collected via the standard log pipeline) and to a Cassini topic (`polar.cert-issuer.events.v1`) for real-time consumption by the observer agent. This dual-emission pattern matches how other Polar services emit telemetry; the cert issuer is not a special case.

## 5. Component Architecture

The Rust binary is organized into the following internal modules. This is a logical decomposition for review; the actual workspace layout can collapse some of these into a single crate if it makes the build simpler.

The **HTTP layer** terminates TLS, parses requests, and dispatches to the issuance handler. It uses `axum` with `tokio` for the async runtime and `rustls` for TLS termination. The cert issuer's own serving certificate is issued by the Smallstep CA via cert-manager's standard ACME or JWK provisioner flow, with a longer lifetime (e.g., 90 days) than anything the cert issuer itself issues. This is the bootstrap-of-the-bootstrap problem, addressed in Section 9.

The **OIDC validation module** fetches and caches JWKS from the Kubernetes API server's OIDC discovery endpoint, validates JWT signatures, and extracts the workload identity claims from the token. It is written against a generic OIDC contract — issuer URL, audience, JWKS endpoint, claim paths — even though v1 only configures one issuer. Writing it generically costs almost nothing and means v2's multi-issuer addition is a configuration change rather than a refactor. JWKS are cached in-memory with TTLs respecting the `Cache-Control` header from the JWKS endpoint, with a default of 5 minutes and a maximum of 1 hour to bound key-rotation propagation. The `jsonwebtoken` crate handles JWT parsing and signature verification.

The configuration for the issuer captures everything the validator needs: the expected `iss` claim, the expected `aud` claim (a required value, not optional, to prevent token replay), the JWKS URL (auto-discovered from the issuer if not specified), the JSON path within the JWT claims that contains the agent's stable identity (for Kubernetes, this is `sub`, which yields `system:serviceaccount:<namespace>:<service-account>`), the JSON path for the per-instance binding identifier (for Kubernetes, this is the pod UID nested under the `kubernetes.io` claim block), and the allowed signing algorithms (RS256, ES256, EdDSA — never HS256 or "none").

Agents must request their projected service account tokens with the audience set to the cert issuer's configured audience. This is a single line in the pod spec's `serviceAccountToken` projected volume source, and it's the kind of thing that's easy to miss and important to get right. The cert issuer rejects tokens with mismatched audience claims with a clear error message identifying the misconfiguration. This is called out explicitly in the agent deployment documentation and surfaced as a distinct telemetry outcome (`INVALID_AUDIENCE`) so misconfigurations are immediately visible.

The **CA client module** wraps the Smallstep CA's API for certificate issuance. It uses the `step-ca` JWK provisioner credentials configured at startup to authenticate to the CA. When an issuance request arrives and the OIDC token has been validated, the module constructs a CSR with the agent's identity (extracted from the JWT) encoded in the Subject Alternative Name. The certificate's `notAfter` is set to `now + configured_lifetime` (default 1 hour, see Section 8). The certificate metadata includes the issuance session ID for downstream audit correlation. CA client errors are distinct from validation failures in the audit log: a successful OIDC validation followed by a CA failure is a different operational signal than a rejected token, and the alerting rules need to distinguish them.

The **telemetry module** emits structured events for every issuance attempt, successful or not. Events are published to stdout as newline-delimited JSON for the standard log pipeline, and to the Cassini topic `polar.cert-issuer.events.v1` for real-time stream consumption. The dual emission is deliberate: stdout is the durable record; Cassini is the low-latency stream. If Cassini is unavailable, events are still in stdout. The module also exposes Prometheus metrics on a separate port (request rates by outcome, JWKS cache hit rates, CA latency histograms, OIDC validation latency histograms).

There is no in-process state module in v1. The cert issuer is stateless: a request comes in, gets validated, results in a CA call, returns a response. No nonces, no ephemeral secrets, no challenge-response, no rate-limit tables. v2 may add some of this for the init-container handshake protocol, but v1 doesn't need it. The absence of state means horizontal replication is trivial — any instance can serve any request — and the deployment can be a standard multi-replica Kubernetes Deployment with no sticky routing or external state store.

This is a meaningful simplification compared to the original attestation service spec, which had a multi-step protocol with nonces and ephemeral HMAC secrets. v1 doesn't have those because v1 doesn't have the threat model that justified them. v1's threat model is "an attacker who steals an agent's projected SA token can request a certificate as that agent." That's true, and it's also true in cert-manager-mounts world (where the attacker just steals the mounted cert directly), so v1 isn't worse than the baseline. v2's challenge-response and HMAC binding mitigate token theft and are part of the v2 hardening pass.

## 6. The Issuance Flow

The v1 flow is a single HTTP request. There's no challenge-response, no multi-step handshake, no ephemeral secret derivation. An agent at startup makes one call:

```
POST /issue
Authorization: Bearer <projected-sa-token>
Content-Type: application/json

{
  "csr": "<PEM-encoded CSR with agent identity in SAN>"
}
```

The cert issuer validates the bearer token against the configured OIDC issuer's JWKS, checks the audience claim matches, extracts the agent's identity from the configured claim path, and verifies that the SAN in the CSR matches that identity. The SAN-vs-identity check prevents an agent from requesting a cert with a different identity than its token authorizes — without it, the git observer's token could be used to obtain a cert in the OCI resolver's name. The cert issuer then forwards the CSR to the Smallstep CA, signed with the configured `notAfter`, and returns the issued certificate chain along with metadata.

```
RESPONSE 200
Content-Type: application/json

{
  "certificate": "<PEM-encoded cert chain>",
  "ca_chain": "<PEM-encoded CA certs>",
  "session_id": "<UUID for audit correlation>",
  "expires_at": "<ISO-8601 timestamp>"
}
```

The init container generates the CSR's private key locally and never transmits it. This is the standard CSR pattern and it's worth being explicit about: the cert issuer never sees the private key, only the CSR (which contains the public key). The private key is written to the shared `emptyDir` volume alongside the issued cert and read by the workload container at startup. If the cert issuer is compromised, the attacker can issue arbitrary certificates but cannot impersonate existing agents whose private keys were generated in their own init containers.

If validation fails at any step — invalid token, mismatched audience, SAN-vs-identity mismatch, CA unavailable — the cert issuer returns a 4xx or 5xx response with a structured error body identifying the specific failure mode. The init container exits non-zero on failure, which prevents the workload container from starting (Kubernetes will not start a workload container after a failed init container). Kubernetes applies the configured restart policy, which gives the init container another chance on the next pod attempt. Hanging or polling indefinitely is wrong; explicit failure with a clean restart is right.

## 7. Agent Integration via the Init Container

v1 uses the `polar-nu-init` init container as the cert handshake client. The init container is a small image — a Rust binary linked against the same OIDC and CSR-generation crates the cert issuer uses internally — that runs to completion before the agent's workload container starts. The agent itself does not link any HTTP client code, does not know the cert issuer's address, and does not hold its own projected SA token at runtime.

A pod that needs an attested certificate has the following shape. It declares a projected service account token volume with the audience set to the cert issuer's configured audience (this is non-negotiable; missing or wrong audience is the most common deployment misconfiguration and is surfaced as a distinct telemetry outcome). It declares an `emptyDir` volume mounted into both the init container and the workload container at a known path (default `/var/run/polar/certs`). It declares the init container with the projected SA token volume mounted, the `emptyDir` volume mounted, and the cert issuer's address provided via env var or downward API. The init container's command runs the handshake: read the SA token, generate an Ed25519 keypair, build a CSR with the expected identity in the SAN, POST to the cert issuer's `/issue` endpoint, write `cert.pem`, `key.pem`, and `ca.pem` to the shared volume, exit zero.

The workload container starts only if the init container exits successfully. It mounts the `emptyDir` volume at the same path, reads the three PEM files at startup, and uses them to construct the mTLS client config it passes to its Cassini connection (and any other mTLS endpoint it talks to). The workload container has no projected SA token volume mounted at all — the SA token is scoped to the init container only. This is enforced by the pod spec, not by trust in the workload container.

The security property this gives you, beyond the configuration cleanup motivation: the projected SA token is the bootstrap credential, and it lives only inside the init container's process for the duration of the handshake (typically well under a second). Once the init container exits, the token is gone. Any code execution vulnerability in the workload container — a deserialization bug in the agent, a malicious dependency, a compromised library — gives the attacker access only to the cert and key on the shared volume, not the bootstrap credential that could be used to obtain new certs indefinitely. The attack surface for token theft drops from "the entire lifetime of the agent" to "the few seconds the init container spends doing one well-defined thing."

The pragmatic property: the agent is unchanged from its current shape. It reads three files at startup and uses them. There's no HTTP client to add, no OIDC validation to integrate, no CSR generation to think about, no error handling for cert issuer unavailability beyond "the file isn't there, exit." The handshake complexity lives in one place — the init container image — and is maintained once.

Renewal is restart-driven. When the cert approaches expiry, the agent's mTLS connections to Cassini fail, its readiness probe fails, Kubernetes restarts the pod, the init container runs again, and the agent comes back up with a fresh cert. There is no in-process renewal path in v1 because the agents have no HTTP client; adding one purely for renewal would defeat the entire point of the init container model. For agents that legitimately cannot tolerate restart-driven renewal — stateful agents with expensive cold-start initialization — the right answer is to either tune the cert lifetime up for those specific agents (per-class lifetime configuration, mentioned in Section 8) or to give those agents an HTTP client and run the handshake in-process. The latter is a deliberate choice for that agent, not a general policy.

The init container's failure modes are worth being explicit about. If the OIDC token validation fails (wrong audience, expired token, unrecognized issuer), the init container exits with a specific non-zero code and logs a structured error identifying the failure. The pod stays in `Init:Error` state and Kubernetes applies the restart policy. If the cert issuer is unavailable (network failure, service down), the init container exits with a different code that distinguishes "transient infrastructure failure" from "deployment misconfiguration." Operators can quickly identify whether they're looking at infrastructure flakiness or a configuration issue from the exit code and log output. If the CA is unavailable but the cert issuer is up, the cert issuer returns a 5xx with a structured body, and the init container surfaces this as a third distinct failure mode. The three failure classes — bad config, cert issuer unreachable, CA unreachable — should be distinguishable from a single glance at `kubectl describe pod`.

## 8. Certificate Lifetime

The default cert lifetime in v1 is 1 hour. This is the same recommendation that came out of the earlier discussion and the reasoning hasn't changed.

A 1-hour default puts the exposure window for a stolen cert in the same order of magnitude as the projected SA token's natural rotation cadence (typically 1 hour, kubelet-managed) and well within Kubernetes' tolerance for routine pod restarts. A 15-minute default would force pod restarts every 15 minutes, which is operationally noisy and not justified by the marginal security gain. A 4-8 hour default would substantially extend the cert's exposure window without a clear corresponding benefit; if agents can't tolerate hourly restarts, that's a robustness problem in the agents that should be fixed independently of cert lifetime.

The lifetime is a configuration parameter, not a code-level constant. Operators can adjust it per deployment. The default is "the first reasonable answer" and the spec acknowledges it should be profiled under real workload conditions. Profiling questions: how long does each agent take to initialize after obtaining its cert? What's the cluster's baseline restart rate from non-cert reasons (deployments, autoscaling, node lifecycle)? How much in-flight work does an agent restart drop? The answers will be different per agent type, and per-agent-class lifetime tuning is supported (the cert issuer can be configured to issue different lifetimes for different requesting identities, though this is an advanced configuration and most deployments won't need it).

## 9. The Cert Issuer's Own Serving Certificate

The cert issuer needs a serving certificate for its own HTTPS endpoint. It cannot issue this to itself — that's a circular dependency. v1 solves this with cert-manager: the cert issuer's serving cert is a standard cert-manager `Certificate` resource backed by a `ClusterIssuer` pointing at the same Smallstep CA, with a longer lifetime (default 90 days) and the cert renewed via cert-manager's standard ACME or JWK provisioner flow. cert-manager mounts the cert into the cert issuer's pod via a Kubernetes secret, just like it does for everything today.

This is a deliberate concession: the cert issuer is the one service that still uses cert-manager mounts, because it can't bootstrap itself. Everything else moves off cert-manager mounts and onto cert-issuer-issued certs. The asymmetry is fine — there's exactly one service that needs a serving cert from outside the cert-issuance flow, and one cert-manager `Certificate` resource for that one service is a much smaller configuration surface than sixteen agents' worth of mounts.

A more elaborate option exists: provision the cert issuer's serving cert from a separate ClusterIssuer with its own root, isolating the cert issuer's serving identity from the CA it uses for agent issuance. This would mean a CA compromise affecting the agent CA wouldn't immediately compromise the cert issuer's serving identity, allowing the cert issuer to keep serving while the agent CA is rotated. This is a v2 hardening direction, not a v1 requirement. v1 uses the same Smallstep CA for both, accepting the coupled-trust property.

## 10. Bootstrap Order

The order in which services come up matters and needs to be explicit. At cluster bring-up, the dependencies are: Smallstep CA must be running before anything else in the Polar stack starts (it's the trust root). cert-manager must be running and configured with a `ClusterIssuer` pointing at the Smallstep CA, so that the cert issuer's serving cert can be provisioned. The cert issuer can then start; it fetches its serving cert from the cert-manager-issued secret, fetches its CA provisioner credential from the configured secrets backend, fetches the OIDC discovery document and JWKS from the Kubernetes API server, and is ready to issue.

Once the cert issuer is up, agents can come up in any order. Each agent's startup includes the cert handshake, which only requires the cert issuer to be available. Cassini comes up like any other agent (it gets its TLS cert from the cert issuer, just like everything else — Cassini is a special agent in that it's the broker, but it's not architecturally privileged in the cert-issuance flow). The credential agent, when it eventually exists, comes up after the cert issuer, gets its cert, and begins serving credentials to clients authenticated via certs from the cert issuer.

The bootstrap doesn't have a circular dependency because the cert issuer's only Polar-internal dependency is the Smallstep CA, and the Smallstep CA is below the Polar stack in the dependency graph (it's external infrastructure shared with cert-manager). cert-manager is also external to the Polar stack. The cert issuer's external dependencies (Kubernetes API server's OIDC issuer, Smallstep CA, secrets backend) are all things that exist independently of the Polar agents.

## 11. Telemetry and Integration with the Observability Framework

Every issuance attempt emits a structured event with the following fields: timestamp, session ID (UUID for audit correlation), workload identity (extracted from the OIDC token), pod UID (extracted from the OIDC token's claims), namespace, OIDC issuer (will be the Kubernetes cluster's issuer in v1, but the field is present so v2's multi-issuer support doesn't change the schema), CSR identity (the SAN from the CSR), validation outcome (`SUCCESS`, `INVALID_TOKEN`, `INVALID_AUDIENCE`, `IDENTITY_MISMATCH`, `CA_UNAVAILABLE`, `INTERNAL_ERROR`), failure reason (human-readable string, only on non-success), latency breakdown (OIDC validation time, CA call time, total time), and the issued certificate's serial number (only on success).

Events go to two destinations. Stdout receives the canonical JSON-line representation, collected by the standard Kubernetes log pipeline. The Cassini topic `polar.cert-issuer.events.v1` receives the same event for stream consumption by the observer agent. This dual emission is the same pattern other Polar services use; the cert issuer is consistent with the framework, not a special case.

The observer agent subscribes to `polar.cert-issuer.events.v1` and forwards events to the processor for ingestion into the Neo4j knowledge graph. The graph schema includes a `CertIssuanceEvent` node type with relationships to `Pod`, `ServiceAccount`, and `Certificate` nodes. The certificate's serial number is the join key between issuance events and downstream mTLS handshake logs, creating a correlation chain from cert use back to issuance. This ingestion is downstream of the cert issuer; the cert issuer publishes fire-and-forget and does not depend on the observer or the graph for any verification decision. The dependency direction is asymmetric, which is what keeps the system acyclic.

Recommended alerts for v1: repeated issuance failures from a single workload identity within a short window (suggests misconfiguration or attack); CA unavailability (a hard failure that blocks all new pod starts and pages on-call); JWKS fetch failures (suggests the Kubernetes API server is unreachable or has rotated keys without grace); audience mismatch failures (suggests a deployment misconfiguration where pod specs aren't requesting the right token audience).

## 12. v2 Directions

This section is not a v2 spec. It documents the directions v1 deliberately defers and the conditions under which they should be added. The point is to make the deferrals intentional rather than accidental — "we'll add it later" is a sentence that often means "we'll never add it" if the trigger conditions aren't written down.

**Cosign signature verification.** Add cosign verification of the requesting agent's image as part of the issuance flow. The init container presents its image digest in the request, and the cert issuer verifies that the digest has a valid cosign signature from the configured pipeline signing key. This adds supply-chain provenance assurance: an unsigned init container image cannot obtain a certificate. **Trigger:** add when a concrete supply-chain threat justifies it (a customer requirement, a security review finding, or a decision that supply-chain assurance is part of the product's security posture). The CI/CD pipeline must be updated to produce cosign-signed images first; the cert issuer's verification is downstream of that pipeline change.

**Workload-image attestation via downward API.** v1's init container verifies its own image (implicitly, when cosign is added) but not the workload container's image. The init container can read the workload container's image digest from the pod's downward API and include it in the cert request, so the cert issuer can verify the cosign signature on the *workload* image as well. This closes the gap where a legitimate init container is paired with an unsigned workload image — a real attack vector if the pod spec can be manipulated by anyone other than the trusted CI/CD pipeline. **Trigger:** ship alongside or after cosign verification; the two compose. Without cosign, workload-image attestation has no verification primitive to invoke.

**Multi-issuer OIDC abstraction.** The OIDC validation module is already written generically; v2 lifts the configuration restriction to a single issuer and supports multiple. The use cases are GitLab OIDC tokens (for GitLab CI workloads requesting Polar certs), SPIFFE/SPIRE-issued JWT-SVIDs (for bare-metal workloads), and possibly AWS IRSA or GCP Workload Identity for hybrid-cloud deployments. **Trigger:** add when there's a concrete second issuer with a real workload that needs a cert. Adding multi-issuer support speculatively, before any second issuer exists, invites scope creep.

**Bare-metal targets.** Bare-metal node attestation is genuinely different from Kubernetes-pod attestation and probably wants TPM quotes or SPIRE node attestation as the identity primitive, not OIDC. **Trigger:** add when there's a concrete bare-metal deployment with hardware that supports TPM-based or SPIRE-based attestation. Don't speculate about the abstraction before the deployment exists.

**Challenge-response handshake with HMAC binding.** The original attestation spec included a multi-step protocol with nonces and HMAC-derived ephemeral secrets to bind the request to a specific instance and prevent replay. v1 omits this because the init container's brief token-exposure window already substantially mitigates the realistic replay scenarios. The challenge-response further hardens against replay even if the token is somehow captured during the init phase. **Trigger:** add if there's a concrete reason to believe SA tokens can be obtained but private keys cannot, which is a fairly specific threat scenario. For most environments, the init container's reduction of the token-exposure window is sufficient.

**Per-agent-class issuance policies.** v1 issues certs based purely on OIDC validation; any agent with a valid token gets a cert with the SAN matching its identity. v2 may introduce policies that constrain which agents can request certs with which SANs at a finer granularity, similar to the (agent identity, credential_id) allowlist planned for the credential agent. **Trigger:** add when there's a concrete policy requirement beyond what the SAN-vs-identity check already enforces. The v1 check already prevents the obvious misuse (one agent's token requesting another agent's identity); v2 policies would be defense-in-depth.

**Sigstore keyless mode.** If cosign verification is added and the CI/CD pipeline adopts Sigstore keyless signing via Fulcio, the cert issuer's verification path shifts from a static public key to verifying against the Fulcio CA root and the Rekor transparency log. This is a real key-management improvement (no long-lived signing key) but depends on the CI/CD pipeline making the change first. **Trigger:** follows the cosign verification trigger.

## 13. Implementation Plan

v1 ships in three phases.

**Phase A: core service and init container.** HTTP layer with axum/rustls, OIDC validation against the Kubernetes cluster's issuer, CA client integration with Smallstep, basic stdout telemetry, Prometheus metrics. The `polar-nu-init` image extended with the cert handshake binary. One agent (recommended: the smallest one, probably one of the simpler observers) updated to use the init container pattern instead of mounting cert-manager certs, as the integration proof. **Acceptance:** the chosen agent's pod with the init container in its spec can obtain a certificate end-to-end, the certificate validates against the Smallstep CA chain, the agent uses the cert for mTLS to Cassini successfully, and a pod with a missing or wrong-audience SA token fails the init container cleanly with a distinct exit code and clear error log.

**Phase B: observability integration.** Cassini topic publication of issuance events, observer agent subscription and graph ingestion, alerting rules for the recommended alerts in Section 11. **Acceptance:** issuance events appear in the knowledge graph with correct relationships, alerting fires correctly on the recommended trigger conditions in a staging environment.

**Phase C: deployment hardening.** Multi-replica deployment of the cert issuer, NetworkPolicy controls, secrets backend integration for the CA provisioner credential (rather than a Kubernetes secret in the same namespace), runbook for cert-manager-issued serving cert renewal failures. **Acceptance:** the cert issuer can survive an instance failure without service interruption, the secrets backend integration works in staging, the runbook is tested.

Phases A and B together constitute the minimum viable v1; Phase C is hardening that should ship before the cert issuer is treated as load-bearing for the rest of the Polar stack. Phase A alone is enough to validate the design end-to-end in a development cluster, but not enough to depend on in production.

After Phase C, the credential agent work begins. The credential agent's design (already specified in its own issue) authenticates clients via certs from the cert issuer, which is why the cert issuer has to be production-ready before the credential agent's Phase 2 starts. Phase 1 of the credential agent issue (cleaning up the git observer's credential loading) is independent and can proceed in parallel with any of the cert issuer phases.

## 14. Risks and Open Questions

The biggest risk is that v1 ships and v2 never gets prioritized, leaving the security improvements (cosign, workload-image attestation, challenge-response) on the shelf indefinitely. Section 12's trigger conditions are the mitigation, but they only work if the team actually checks them periodically. The recommendation is to schedule a v2 review six months after v1 ships, with the trigger conditions as the agenda. If none of the triggers have fired, that's a real signal that v2 isn't currently justified, and the deferral becomes intentional. If triggers have fired and v2 hasn't started, that's a real signal that v2 is overdue.

A secondary risk is the rollout cost of switching every agent's pod spec to use the init container pattern. The Dhall manifest infrastructure has to learn the init container shape (volumes, audience-scoped projected SA token, init container declaration), and every agent's manifest has to be updated. This is a one-time cost but it's a real cost, and it interacts with the configuration drift problem the cert issuer is meant to solve: while the rollout is in progress, you have a mix of cert-manager-mount agents and cert-issuer-init agents, which is more drift, not less, until the migration completes. Mitigation: pick one agent type to migrate first (Phase A's acceptance criterion), validate the pattern end-to-end, then roll the rest in a single batch with a clear cutover plan rather than dribbling them in one at a time.

A third risk is that the init container becomes a place where ad-hoc startup logic accumulates. The credential agent issue mentions `polar-nu-init` as the bootstrap mechanism for that service too, with additional responsibilities (exposing the projected SA token to the credential agent processor for Kubernetes API access). As more services depend on the init container for various pre-flight tasks, the init container's scope grows and it becomes harder to reason about what it does in any given pod. Mitigation: keep the cert handshake logic in its own binary inside the init container image, with a clear interface, and treat any other pre-flight tasks as separate binaries that the init container's command sequence invokes. The init container is a container image, not a single program; multiple small programs with clear responsibilities are easier to maintain than one big program that does everything.

Open question: the format of the issuance request. v1 uses a CSR-based flow where the init container generates the keypair locally and submits a CSR. An alternative is the cert issuer generating the keypair and returning both cert and key, which is simpler but means the cert issuer holds (briefly) every workload's private key in memory. The CSR-based flow is the right answer for security reasons (the cert issuer never sees private keys), and it's not a friction point in the init container model because the CSR generation logic lives in one place — the init container image — rather than being repeated across every agent. Recommendation: stick with CSR.

Open question: how does the cert issuer authenticate to the Smallstep CA? The CA's JWK provisioner expects a JWT with a specific signing key. v1 stores this key in a Kubernetes secret in the cert issuer's namespace, with restricted access. v2 should move this to the same secrets backend used for any other sensitive material (Vault, KMS-backed envelope encryption). This is a deployment-hardening question, not a protocol question, and the choice depends on what secrets backend the cluster operates.

Open question: should the cert issuer support cert revocation? v1 doesn't, because short cert lifetimes (1 hour) make revocation unnecessary in most scenarios — a stolen cert is naturally invalidated within an hour. CRL or OCSP support is a v2+ direction if there's a concrete need to invalidate certs faster than their natural expiry. Smallstep CA supports both; the cert issuer would just need to surface the capability. Most environments won't need this; defer until asked.
